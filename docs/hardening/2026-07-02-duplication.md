### 1. Migration filename version parsing `<version>_<name>.up.sql` — score 60

- **Kind:** body-similarity (+ stdlib-shadow: all copies are hand-rolled `strings.Cut`)
- **Locations:** `internal/migrate/runner.go:81` (`parseFilename`), `internal/migrate/create.go:53` (inside `maxVersion`), `internal/codegen/generate.go:137` (`leadingVersion`)
- **Copies:** 3 × ~10 LOC across 2 packages
- **Risk flags:** `divergent`, indirectly-tested (`parseFilename` via runner integration test, `maxVersion` via `TestCreateSequential`), **untested-copy** (`leadingVersion`/`readUpMigrations` has zero coverage)
- **Divergence (real behavioral gap):** `parseFilename` errors on malformed names (caller skips), `maxVersion` silently skips, but `leadingVersion` swallows `ParseUint` errors and returns **0** — a malformed `.up.sql` that golang-migrate would never apply still gets included FIRST in codegen's schema build, so generated models can silently diverge from the real schema.
- **Verifier: CONFIRMED.** The writer (`create.go:91` builds `version + "_" + slug`), reader, and codegen consumer each re-implement one domain contract the `internal/migrate` package owns. Verifier corrections: drop the two DSN-scheme sites the finder lumped in (`db.go:22`, `repl.go:51` — different intent, and `strings.Cut` is not a drop-in there because the `i > 0` check also rejects an empty scheme); reject the `internal/migname` micro-package idea.
- **Final action:** CONSOLIDATE into `internal/migrate`: export `ParseFilename(base string) (uint64, string, error)` next to `Create` (the format owner), implemented with `strings.Cut`. `loadNames` keeps skip-on-error; `maxVersion` calls it and skips on error; codegen's `readUpMigrations` calls it and **propagates (or at minimum skips) errors instead of defaulting to version 0**.
- **Follow-up:** `/refactor:extract internal/migrate.ParseFilename — unify runner.go:81, create.go:53, codegen/generate.go:137; fix leadingVersion version-0 swallow; add unit tests for malformed names`

### 5. In-place filter loop — score 16

- **Kind:** stdlib-shadow
- **Locations:** `internal/codegen/catalog.go:131` (`Table.dropColumn`), `internal/codegen/catalog.go:154` (`Catalog.applyDrop`)
- **Copies:** 2 × 8 LOC, same package. **Risk flags:** tested (dropColumn path only via `catalog_test.go:16`; **applyDrop DROP TABLE path has no test coverage**)
- **Evidence:** both use the `out := s[:0]; for … append` in-place filter idiom to drop entries by name. `slices.DeleteFunc` (Go ≥1.21) does this in one line and additionally zeroes the freed tail — clears stale `*Table` pointers for GC, unobservable otherwise.
- **Verifier: CONFIRMED** — true drop-in on Go 1.26.1 (same nil/empty handling, order preservation, all-matches removal, in-place reuse); only the predicate inverts (keep `!=` → delete `==`).
- **Final action:** STDLIB-REPLACE → `slices.DeleteFunc` at both sites; delete the manual loops.
- **Follow-up:** inline stdlib rewrite (no new package): `t.Columns = slices.DeleteFunc(t.Columns, func(c Column) bool { return c.Name == name })` and the `[]*Table` equivalent in `applyDrop`; add a DROP TABLE test while touching it.
